## Phase 1: Foundation & Protobuf Schema

### Deliverables
- `protos/pouw/v1/pouw.proto` - Complete schema v1
- iOS Swift bindings via `swift-protobuf`
- Android Kotlin bindings via `protobuf-kotlin`
- Cross-platform test vectors

### Protobuf Schema (v1)

```protobuf
syntax = "proto3";
package pouw.v1;

option swift_prefix = "PoUW";

// Challenge Token from node to client
message ChallengeToken {
  uint32 version = 1;
  bytes node_id = 2;           // 32 bytes
  bytes task_id = 3;           // 16-32 bytes
  bytes challenge_nonce = 4;   // 16-32 bytes
  int64 issued_at = 5;         // Unix timestamp (seconds)
  int64 expires_at = 6;        // Unix timestamp (seconds)
  Policy policy = 7;
  bytes node_signature = 8;    // Ed25519 or Dilithium signature
}

// Policy for receipt generation
message Policy {
  uint32 max_receipts = 1;
  uint64 max_bytes_total = 2;
  uint64 min_bytes_per_receipt = 3;
  repeated ProofType allowed_proof_types = 4;
}

// Proof types supported
enum ProofType {
  PROOF_HASH = 0;
  PROOF_MERKLE = 1;
  PROOF_SIGNATURE = 2;
}

// Signature schemes
enum SignatureScheme {
  ED25519 = 0;
  DILITHIUM2 = 1;
  DILITHIUM3 = 2;
  DILITHIUM5 = 3;
}

// Receipt generated by client after verification
message Receipt {
  uint32 version = 1;
  bytes task_id = 2;           // From ChallengeToken
  string client_did = 3;       // did:zhtp:...
  ProofType proof_type = 4;
  uint64 bytes_verified = 5;
  bool result_ok = 6;
  int64 started_at = 7;        // Unix timestamp (ms)
  int64 finished_at = 8;       // Unix timestamp (ms)
  bytes challenge_nonce = 9;   // From ChallengeToken (binding)
  bytes receipt_nonce = 10;    // Unique per receipt (16-32 bytes)
  bytes aux = 11;              // Optional: merkle path, sig data, etc.
}

// Signed receipt
message SignedReceipt {
  Receipt receipt = 1;
  SignatureScheme sig_scheme = 2;
  bytes signature = 3;         // Detached signature over receipt bytes
}

// Batch submission
message ReceiptBatch {
  uint32 version = 1;
  string client_did = 2;
  bytes batch_nonce = 3;       // Unique per batch (16-32 bytes)
  repeated SignedReceipt receipts = 4;
}

// Submission response (from node)
message SubmissionResponse {
  repeated bytes accepted = 1;  // List of receipt_nonces accepted
  repeated Rejection rejected = 2;
  int64 server_time = 3;
}

message Rejection {
  bytes receipt_nonce = 1;
  RejectionReason reason = 2;
}

enum RejectionReason {
  UNKNOWN = 0;
  EXPIRED = 1;
  REPLAY = 2;
  POLICY = 3;
  BAD_SIGNATURE = 4;
  BAD_PROOF = 5;
}
```

### Build Configuration

**iOS:** Add to Podfile:
```ruby
pod 'SwiftProtobuf', '~> 1.0'
```

**Android:** Add to `app/build.gradle`:
```gradle
plugins {
    id "com.google.protobuf" version "0.9.4"
}

dependencies {
    implementation 'com.google.protobuf:protobuf-kotlin-lite:4.28.0'
}

protobuf {
    protoc { artifact = 'com.google.protobuf:protoc:4.28.0' }
    plugins {
        kotlin { artifact = 'com.google.protobuf:protoc-gen-kotlin:4.28.0:jvm8' }
    }
    generateProtoTasks {
        all().configureEach { task ->
            task.builtins { java { option 'lite' } }
            task.plugins { kotlin {} }
        }
    }
}
```

## Phase 2: iOS PoUW Client Implementation

### Architecture

```
ios/
├── PoUW/
│   ├── PoUWController.swift      # Main orchestrator
│   ├── VerifierEngine.swift      # Hash/Merkle/Sig verification
│   ├── IdentitySigner.swift      # DID-based signing (Secure Enclave)
│   ├── ReceiptStore.swift        # Persistent queue (Core Data)
│   ├── SubmissionClient.swift    # Network layer
│   └── Models/
│       ├── ReceiptState.swift    # State machine enum
│       └── PoUWError.swift       # Error types
└── RCTPoUW.swift                 # RN Bridge Module
```

### Module Details

#### 1. VerifierEngine
```swift
protocol VerifierEngineProtocol {
    func verifyHash(bytes: Data, cidDigest: Data) -> Bool
    func verifyMerkle(leaf: Data, proof: MerkleProof) -> Bool
    func verifySignature(message: Data, signature: Data, publicKey: Data) -> Bool
}
```

#### 2. IdentitySigner
- Leverages existing `Identity` infrastructure
- Uses Dilithium5 via LibOQS (already integrated)
- Keys never leave Rust FFI boundary

#### 3. ReceiptStore (Core Data)
```swift
@Model
class ReceiptEntity {
    var receiptNonce: Data
    var taskId: Data
    var state: ReceiptState
    var signedReceiptData: Data
    var createdAt: Date
    var retryCount: Int
    var lastError: String?
}
```

#### 4. SubmissionClient
- Uses existing `NativeZhtpApi` for QUIC/HTTP transport
- Endpoints: `GET /pouw/challenge`, `POST /pouw/submit`

#### 5. PoUWController API
```swift
protocol PoUWControllerProtocol {
    func verifyAndRecord(contentId: Data, bytes: Data, providerId: Data?) async throws
    func flushReceipts() async throws
    func getPendingCount() -> Int
    func getStats() -> PoUWStats
}
```

## Phase 4: React Native Bridge

### Strict RN Boundary (Spec C)

RN sees **exactly** this interface:

```typescript
// src/native/PoUW.ts
export interface PoUW {
  verifyContent(
    contentId: Uint8Array,
    bytes: Uint8Array,
    providerId?: Uint8Array
  ): Promise<void>;

  flush(): Promise<void>;
  
  getPendingCount(): Promise<number>;
}
```

**Hard rules:**
- RN never passes URLs
- RN never sees keys
- RN never sees receipts or signatures
- RN never serializes protobuf
- RN never performs cryptography
- RN never talks directly to reward endpoints
- **RN is a button + lifecycle trigger, nothing more**

### Native Module Registration

**iOS (RCTPoUW.swift):**
```swift
@objc(RCTPoUW)
class RCTPoUW: NSObject, RCTBridgeModule {
  static func moduleName() -> String! { "PoUW" }
  
  @objc func verifyContent(...)
  @objc func flush(...)
}
```

**Android (PoUWModule.kt):**
```kotlin
class PoUWModule(reactContext: ReactApplicationContext) : 
    ReactContextBaseJavaModule(reactContext) {
  override fun getName() = "PoUW"
  
  @ReactMethod
  fun verifyContent(...)
  
  @ReactMethod
  fun flush(...)
}
```

## Phase 6: Integration Testing & Production Hardening

### Cross-Platform Parity Matrix

| Test | iOS | Android | Node |
|------|-----|---------|------|
| Same protobuf serialization | ✅ | ✅ | ✅ |
| Same receipt signature | ✅ | ✅ | ✅ |
| Challenge binding works | ✅ | ✅ | ✅ |
| Batch submission accepted | ✅ | ✅ | - |
| State machine behavior | ✅ | ✅ | - |
| Offline queue persistence | ✅ | ✅ | - |

### E2E Test Flow
```
1. Mobile requests challenge from node
2. Node returns ChallengeToken
3. Mobile verifies content (hash/merkle/sig)
4. Mobile generates SignedReceipt
5. Mobile batches receipts
6. Mobile submits batch to node
7. Node validates and responds
8. Mobile processes accept/reject
```

### Security Checklist
- [ ] Private keys never exported from Secure Enclave/Keystore
- [ ] All signing happens in native code
- [ ] RN never sees keys, receipts, or signatures
- [ ] Protobuf serialization is deterministic
- [ ] Nonce uniqueness enforced (24h+ window)
- [ ] Replay protection verified

## Blockers & Dependencies

### External Dependencies
1. **Node endpoints** (`/pouw/challenge`, `/pouw/submit`) - Backend #875
2. **Identity system** - Already available via Rust FFI
3. **QUIC transport** - Already available via `NativeZhtpApi`/`NativeQuicBridge`

### Potential Blockers
1. **Ed25519 vs Dilithium**: Clarify which signature scheme for receipts
2. **Merkle proof format**: Need spec from backend for proof structure
3. **Rate limiting values**: Need concrete values for policy enforcement
4. **Test node endpoint**: Need dev endpoint for testing

## Success Criteria

- [ ] iOS and Android produce **identical** receipt bytes for same input
- [ ] Both platforms' receipts are accepted by node validation
- [ ] Offline queue persists across app kills
- [ ] Battery drain ≤ 10% for 1 hour continuous verification
- [ ] No PII in receipts (no URLs, query strings, content)
- [ ] Security audit passes

## Notes

### Existing Infrastructure to Leverage
- **iOS**: `LibOQSSIG.swift`, `ZhtpClient.swift`, `NativeZhtpApi.swift`
- **Android**: `Identity.kt`, `NativeQuicBridge.kt`, `NativeZhtpApi.kt`
- **Both**: QUIC/HTTP transport already implemented
- **Crypto**: Dilithium5 + Ed25519 already available via LibOQS

### Critical: Cross-Platform Parity
> "If Android and iOS receipts differ → rewards break."

This is the #1 priority. Use shared:
- Protobuf schema
- Deterministic serialization
- Canonical test vectors
- Shared test fixtures
